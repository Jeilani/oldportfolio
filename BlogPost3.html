Priortize Documentation/h1>

Posted on April 9, 2019

One of the more emphasized lessons I heard in the beginning of my journey proved to be incredibly valuable. Document where you should.

For a while I was too arrogant to take that piece of advice into serious consideration. I mean how hard can it be to decipher your own code? Turns out, just like class notes written in bad handwriting, code becomes hard to decipher if you spend enough time away from it.

Obviously you don't want to document self-explantory code. Often times, the legibility of code can be handled by naming variables, objects, functions, and methods appropriately. Documentation plays an incredibly critical role when that strategy reaches its limits.

A great example is in an enviornment like Processing animation projects. There are plenty of opportunities within the HTML canvas where you find yourself geting into mathematical expressions that aren't self-evident. In which case, document! You'll thank yourself later.
